🌍 Le marché FP&A en Afrique de l'Ouest : Un désert à saisir
La réponse courte : Non, il n'existe pas de véritable solution FP&A dédiée aux PME d'Afrique de l'Ouest.

Ce qui existe réellement
1. Des logiciels de comptabilité locaux (pas du FP&A)
AFRI-COMPTA (distribué par Ayecom Consulting) couvre la comptabilité générale, auxiliaire, analytique et budgétaire, avec une documentation et un paramétrage adapté aux pays SYSCOHADA/SYSCOA d'Afrique de l'Ouest. Son module AFRI-BUDGET permet de planifier et suivre des budgets linéaires ou saisonniers. CloudZero C'est de la gestion budgétaire basique — loin du FP&A stratégique (forecasting, scenario planning, KPIs).
2. Des ERP locaux avec modules financiers
SmartERP de WEBGRAM (Dakar, Sénégal) centralise la comptabilité, la gestion commerciale, les stocks et la trésorerie, avec une conformité aux normes SYSCOHADA, adapté au Sénégal, Côte d'Ivoire, Cameroun, Togo, Gabon. DNA Growth Mais c'est un ERP, pas un outil FP&A dédié à la planification et l'analyse stratégique.
3. Des solutions internationales inadaptées
Les logiciels internationaux « standards » ne prennent pas toujours en compte les réglementations locales strictes (BCEAO, BRVM, normes OHADA), ni les logiques métiers spécifiques à la microfinance ou aux coopératives très présentes dans la région. Cube
Des outils comme Odoo permettent une certaine personnalisation pour les contextes fiscaux du Sénégal, du Cameroun ou du Bénin, mais exigent un accompagnement technique solide. Jirav

Pourquoi ce vide existe-t-il ?
L'avenir de la gouvernance d'entreprise en Afrique de l'Ouest repose sur des outils flexibles, adaptés aux normes OHADA et capables d'intégrer l'intelligence artificielle prédictive — mais ce marché reste largement sous-équipé. SAP
Les freins structurels sont clairs :
Côté marché : Les PME africaines sont peu habituées à payer pour un logiciel de pilotage financier. La fonction FP&A est souvent inexistante ou assurée par le DAF seul avec Excel.
Côté offre : Les éditeurs mondiaux (Anaplan, Planful) ignorent ce marché car les tickets moyens sont trop faibles. Les acteurs locaux se concentrent sur la comptabilité réglementaire (SYSCOHADA) et non sur l'analyse prospective.
Côté infrastructure : Selon la BCEAO, plus de 65% des transactions financières dans l'UEMOA passent par des canaux numériques en 2025 Cube — le digital s'accélère, mais les outils de pilotage stratégique n'ont pas suivi.

🔍 Ce que le diagramme fait bien
Complétude métier — Les 6 couches couvrent l'essentiel du cycle FP&A : revenus → coûts → états financiers → pilotage. C'est cohérent avec ce qu'on voit dans le fichier Excel.
Traçabilité — Chaque entité est rattachée à Organization + Period, ce qui garantit l'isolation multi-tenant et la granularité temporelle.
Les 3 statements sont liés — IncomeStatement, CashFlowStatement et BalanceSheet existent comme entités distinctes mais partagent la même Period. C'est fidèle à la logique du modèle Excel.

⚠️ Ce que je changerais ou questionnerais
1. Trop de classes "calculées" en base de données
IncomeStatement, CashFlowStatement, BalanceSheet sont des vues calculées, pas des entités à persister. En pratique, ces états se recalculent dynamiquement à partir des données sources. Les stocker en base crée un risque de désynchronisation.

Recommandation : Les transformer en Views ou services de calcul, pas en tables.


2. Client trop simpliste pour une PME multi-secteurs
On n'a que name et sector. Pour gérer les créances clients correctement, il faut ajouter : payment_terms (délai de paiement en jours), credit_limit, billing_currency. C'est critique pour le module Cash Flow.

3. La relation Scenario → Budget est incomplète
Un scénario ne se contente pas de modifier un budget — il doit pouvoir surcharger des hypothèses au niveau des drivers (prix, volume, taux de croissance). La classe ScenarioHypothesis existe mais n'est pas reliée aux entités concrètes qu'elle impacte (RevenueAccrued, EmployeeCost, etc.).

4. Pas de RevenueDriver
Le fichier Excel montre que le revenu est piloté par client × période. Mais en pratique une PME a des drivers plus fins : prix unitaire, volume, taux de conversion. Sans cette classe, le forecasting reste statique.

5. BankAccount a disparu en V2
Dans la V1 on avait BankAccount avec la gestion Mobile Money (Wave, Orange Money) — c'est une spécificité Afrique de l'Ouest essentielle. Elle a été absorbée dans CashPosition mais devrait rester une entité à part entière.

6. Pas de AuditLog
Pour une application financière, toute modification doit être tracée : qui a changé quoi, quand, quelle valeur avant/après. C'est une exigence de conformité et de confiance pour le DAF.

3. RevenueDriver remplace RevenueAccrued — Le revenu n'est plus une valeur brute mais le produit de drivers : unit_price × volume × growth_rate. Cela permet l'analyse de sensibilité ("si le prix baisse de 10%, quel impact ?"). C'est la base du forecasting intelligent.

4. ScenarioHypothesis reliée aux entités cibles — Via ScenarioTargetEntity (enum), chaque hypothèse pointe maintenant vers une entité concrète : REVENUE_DRIVER, EMPLOYEE_COST, SGA, TAX_RATE, etc. Le scénario impacte réellement le modèle au lieu de flotter dans le vide.

5. BankAccount réintégré — Avec BankAccountType qui inclut nativement WAVE, ORANGE_MONEY, MTN_MOMO. La CashPosition et CashFlowForecastLine sont maintenant liées à un compte bancaire précis. Différenciation forte vs les solutions mondiales.

6. FinancialSnapshot marqué <<computed view>> — Les 3 états financiers ne sont plus des tables persistées mais une vue calculée à la demande, reconstruite à partir des données sources. Zéro risque de désynchronisation.

🎯 Les 5 décisions clés justifiées

1. PostgreSQL plutôt que MongoDB
Les données financières exigent ACID compliance absolue. Un budget ne peut pas avoir des écritures partiellement enregistrées. MongoDB, orienté document, n'offre pas cette garantie au niveau transactionnel. De plus, le Row-Level Security de PostgreSQL isole les données multi-tenant au niveau de la base elle-même — pas juste dans le code.
2. Python pour le moteur de calcul
Node.js utilise des floats IEEE 754 pour les nombres — catastrophique en finance (0.1 + 0.2 = 0.30000000000000004). Python avec la bibliothèque Decimal garantit une précision exacte. Pandas et NumPy accélèrent les calculs matriciels sur 12 périodes × 40 classes du diagramme.
3. NestJS pour les services métier
TypeScript natif partagé avec le frontend, architecture modulaire qui correspond exactement à nos microservices (AuthService, BudgetService, ScenarioService...), et injection de dépendances qui facilite les tests unitaires — indispensable pour un produit financier.
4. DigitalOcean Lagos plutôt qu'AWS Europe
Un utilisateur à Dakar ou Abidjan qui interroge un serveur à Paris subit 180–220ms de latence. Avec le datacenter de Lagos, on tombe à 20–35ms. Sur une connexion 4G instable, cette différence transforme l'expérience utilisateur.
5. BullMQ pour les imports asynchrones
Un import de 5 000 lignes Excel avec recalcul complet peut prendre 8–15 secondes. Bloquer la requête HTTP pendant ce temps est inacceptable. BullMQ permet de retourner immédiatement un job_id, puis de notifier via WebSocket quand c'est terminé — expérience fluide garantie.

  - Vérifie que le hash existe dans Redis

  - Génère un nouveau access_token (8h)
  - Rotation du refresh_token → nouveau token, ancienne clé Redis supprimée
  - Retourne : { access_token, refresh_token }

POST /api/v1/auth/logout

  abstract findOne(id: string, orgId: string): Promise<T>

  abstract create(data: Partial<T>): Promise<T>

  abstract softDelete(id: string, orgId: string): Promise<void>

  // Helper pagination réutilisable

      skip: (page - 1) * limit,

─── hooks/useDebounce.ts ────────────────────────────────────────────────────

function useDebounce<T>(value: T, delay = 400): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);
  return debounced;
}

export function formatFCFA(amount: string | number, withSuffix = true): string {
  const num = typeof amount === 'string' ? parseFloat(amount) : amount;
  const formatted = new Intl.NumberFormat('fr-SN', {
    minimumFractionDigits: 0,
    maximumFractionDigits: 0,
  }).format(num);
  return withSuffix ? `${formatted} FCFA` : formatted;
}

export function parseFCFA(formatted: string): string {
  // Retourne string Decimal — jamais float
  return formatted.replace(/[^\d,-]/g, '').replace(',', '.');
}

export const DEFAULT_CURRENCY = 'XOF';

Règle : toujours passer par formatFCFA() pour afficher un montant

Formatter dates en français :

export function formatDate(
  date: string | Date,
  style: 'full' | 'short' | 'month-year' = 'full'
): string {
  const d = typeof date === 'string' ? new Date(date) : date;
  const options: Record<string, Intl.DateTimeFormatOptions> = {
    full:       { day: 'numeric', month: 'long',  year: 'numeric' },
    short:      { day: 'numeric', month: 'short', year: 'numeric' },
    'month-year': {               month: 'long',  year: 'numeric' },
  };
  return new Intl.DateTimeFormat('fr-SN', {
    ...options[style],
    timeZone: 'Africa/Dakar',
  }).format(d);
}

export function isDateInPeriod(date: Date, start: Date, end: Date): boolean {
  return date >= start && date <= end;
}

@Process('process-excel')
async processExcel(job: Job<ImportJobPayload>) {
  try {
    // 1. Logger début job [SBD-10]
    this.logger.log(`Import job ${job.id} started`);

    // 2. Mettre à jour statut → PROCESSING
    await this.importsRepository.updateStatus(job.data.job_id, 'PROCESSING');

    // 3. Appeler CalcEngine Python via HTTP interne
    //    CalcEngine lit le fichier depuis S3 directement
    //    Jamais passer le fichier en mémoire entre services
    const result = await this.calcEngineClient.post('/import/process', {
      job_id:    job.data.job_id,
      s3_key:    job.data.s3_key,
      period_id: job.data.period_id,
      org_id:    job.data.org_id,
    });

    // 4. Mettre à jour statut → DONE
    await this.importsRepository.updateStatus(job.data.job_id, 'DONE', {
      rows_inserted: result.inserted,
      rows_skipped:  result.skipped,
      completed_at:  new Date(),
    });

    // 5. Créer audit_log [SBD-10]
    await this.auditService.log({
      action:      AuditAction.IMPORT_DONE,
      entity_type: 'IMPORT_JOB',
      entity_id:   job.data.job_id,
      org_id:      job.data.org_id,
      user_id:     job.data.user_id,
    });

    // 6. Notifier via Socket.io
    this.eventsGateway.emit('IMPORT_DONE', {
      job_id:   job.data.job_id,
      inserted: result.inserted,
      skipped:  result.skipped,
    });

    // 7. Publier recalcul KPIs
    await this.calcQueue.add('recalc-kpis', {
      org_id:    job.data.org_id,
      period_id: job.data.period_id,
    });

    this.logger.log(`Import job ${job.id} done — inserted: ${result.inserted}`);

  } catch (error) {
    // 8. Mettre à jour statut → FAILED [SBD-13]
    await this.importsRepository.updateStatus(job.data.job_id, 'FAILED', {
      error_report: { message: error.message },
    });
    // Logger erreur complète côté serveur
    this.logger.error(`Import job ${job.id} failed`, error.stack);
    // Notifier l'échec via Socket.io
    this.eventsGateway.emit('IMPORT_FAILED', { job_id: job.data.job_id });
    throw error; // BullMQ gère les retries
  }
}

from decimal import Decimal, ROUND_HALF_UP, InvalidOperation

def to_decimal(value) -> Decimal:
    """Convertit toute valeur en Decimal sécurisé."""
    if isinstance(value, float):
        # Passer par string pour éviter les erreurs de précision float
        return Decimal(str(value))
    try:
        return Decimal(str(value))
    except InvalidOperation:
        return Decimal('0')

def safe_divide(numerator: Decimal, denominator: Decimal,
                precision: int = 2) -> Decimal:
    """Division sécurisée — retourne 0 si dénominateur nul."""
    if denominator == Decimal('0'):
        return Decimal('0')
    result = numerator / denominator
    return result.quantize(Decimal(10) ** -precision, rounding=ROUND_HALF_UP)

def format_fcfa(amount: Decimal) -> str:
    """Formate un montant en FCFA pour les logs."""
    return f"{amount:,.0f} FCFA"

SYSCOHADA_CLASSES = {
    '1': 'Comptes de ressources durables',
    '2': 'Comptes d\'actif immobilisé',
    '3': 'Comptes de stocks',
    '4': 'Comptes de tiers',
    '5': 'Comptes de trésorerie',
    '6': 'Comptes de charges',
    '7': 'Comptes de produits',
    '8': 'Comptes des autres charges et produits',
}

def is_valid_syscohada(code: str) -> bool:
    import re
    if not re.match(r'^\d{6,8}$', code):
        return False
    return code[0] in SYSCOHADA_CLASSES

def get_line_type(code: str) -> str:
    """Détermine REVENUE ou EXPENSE depuis le code SYSCOHADA."""
    if code.startswith('7'):
        return 'REVENUE'
    elif code.startswith('6'):
        return 'EXPENSE'
    return 'OTHER'

      sub:    payload.sub    ?? 'test-user-id',
      org_id: payload.org_id ?? 'test-org-id',
      role:   payload.role   ?? UserRole.FPA,
      email:  payload.email  ?? 'test@test.sn',
    },
    process.env.JWT_SECRET,
    { algorithm: 'HS256', expiresIn: '8h' }
  );
}
